# 输出、命名与最小验证脚本

## 何时读取本文件

当你需要决定这些问题时，再读取本文件：

- 教程示例应该做成单文件还是拆分文件
- 迁移文件和验证脚本应该怎么命名
- 验证脚本应该保存到哪里
- 最小验证脚本必须包含哪些内容
- 执行失败时应该如何汇报

## 文档风格最小迁移

如果用户给的是官方文档、教程、博客里的简单最小复现，尤其是：

- 单个 `@triton.jit` kernel
- 简单 `elementwise` / `Vector-only`
- 原始代码已经是 1D grid
- 没有 `coreDim` / `ub overflow` / 复杂布局问题

默认按“最小 diff 迁移”：

1. 新增 `import torch_npu`
2. 删除 GPU 专属设备获取逻辑
3. 删除只服务于 GPU 设备对象的断言
4. 将 `device='cuda'` 改为 `device='npu'`
5. 其余逻辑尽量不动

如果用户明确说了这些信号词：

- 官方文档风格
- 严格最小迁移
- 最小 diff
- 不要工程增强版
- 只参考官方迁移示例

则进一步约束：

- 代码只做必要修改
- 优化说明保持极简
- 不要默认加入 `contiguous()`、大量断言、重命名、工程化封装
- 验证脚本也保持最小可运行，不写成测试框架

## 单文件还是拆分文件

### 默认优先单文件

如果原输入本身就是 `test1.py` 这种单文件教程示例：

- 默认优先生成单文件、可直接运行的迁移示例
- 文件结构尽量和原脚本保持一致：`import`、kernel、wrapper、文件底部直接构造输入并运行
- 不要默认拆成“实现文件 + 验证文件”两份

### 适合拆分的情况

只有在这些场景下，才优先拆分：

- 用户明确要求拆分
- 原工程本身已经是分层结构
- 单独复用迁移实现的需求明显高于教程演示需求

如果已经生成了拆分版，但用户要求“按原例子改”，应回退到单文件教程风格。

## 命名与保存规则

每次迁移都要输出一个最小验证脚本，并真正保存到磁盘。

默认保存位置：

1. 优先保存在待迁移脚本所在目录
2. 如果用户直接粘贴代码且指定了目录，就保存到该目录
3. 如果用户直接粘贴代码但没指定目录，就保存到当前任务最相关的位置

推荐命名：

- 原文件：`foo.py`
- 迁移实现：`foo_triton_ascend.py`
- 最小验证脚本：`foo_triton_ascend_example.py`

可接受替代：

- `foo_npu.py`
- `foo_verify.py`
- `foo_triton_ascend_verify.py`

默认优先：

- 单文件教程风格：`<原文件名去后缀>_triton_ascend_example.py`
- 分拆风格：实现文件 `<原文件名去后缀>_triton_ascend.py`
- 分拆风格：验证文件 `<原文件名去后缀>_triton_ascend_example.py`

如果同名冲突，可以追加后缀，但要保留原文件 stem，便于一眼看出来源。

## 最小验证脚本的最低要求

最小验证脚本至少应满足：

- 能直接 `python xxx.py` 运行
- 显式导入 `torch_npu`
- 在 `npu` 设备上构造输入
- 调用迁移后的 kernel 或 wrapper
- 使用 PyTorch reference 对比
- 至少输出 `allclose`、最大误差或等价信息

建议至少覆盖：

- 一组基础输入
- 一组非整除 block 的输入
- 关键 dtype 或一组更大输入

## 失败时怎么处理

不要只生成验证脚本，必须实际执行一次。

如果执行失败：

- 保留已生成的迁移代码和验证脚本
- 输出真实报错
- 说明失败是在环境依赖、编译、还是语义层
- 给出最小排障建议

不要把失败情况包装成“理论上可运行”。

## 结果里必须明确说明的事项

最终回答里至少明确：

- 生成了哪些文件
- 文件保存路径
- 是否实际执行过验证脚本
- 执行命令
- 核心结果，例如 `allclose`、最大误差或失败信息

````markdown
- 如果脚本未跑通，明确卡在哪一步
````
