# 输入方式与上下文补齐

## 何时读取本文件

当你需要判断下面这些问题时，再读取本文件：

- 用户是给了文件路径，还是直接在对话中粘贴代码
- 用户说“迁移这段代码”，但没有明确是整文件还是局部片段
- 文件内容和用户当前贴出的代码不一致
- 代码片段缺少 shape、dtype、wrapper 或调用入口

## 两种输入方式

### 1. 文件路径输入

常见形式：

- “迁移 `foo.py`”
- “读取 `bar.py` 里的这段 kernel 帮我迁移”
- “看这个文件第 N 段代码”

默认处理：

1. 读取用户指定文件
2. 如果用户指定了局部代码，就优先围绕那段迁移
3. 如果局部代码缺上下文，再回看同文件中的相关定义、wrapper、shape/dtype 来源
4. 如果原文件本身是完整教程示例，优先保留原文件结构和风格

默认输出：

- 迁移后的实现
- 保存在原脚本目录下的最小验证脚本
- 实际执行结果或失败信息

### 2. 直接粘贴代码输入

常见形式：

- 用户直接粘贴一个 `@triton.jit` kernel
- 用户贴一段 wrapper
- 用户贴完整脚本并要求迁移

默认处理：

1. 先把当前粘贴内容当作权威输入
2. 判断它是完整脚本、局部片段，还是单个 kernel
3. 如果只是局部片段，也先尽量完成语义等价迁移
4. 如果缺少运行所需上下文，再最小化补齐

默认输出：

- 在回答中直接给出迁移后的代码
- 把最小验证脚本保存到用户指定目录
- 如果用户没指定目录，保存到当前任务最相关的位置
- 明确告诉用户保存路径

## 路径输入与粘贴输入同时出现

如果用户同时给了文件路径和当前消息中的代码片段：

1. 以用户明确指定“要迁移的那段代码”为准
2. 如果片段与文件不一致，优先按当前消息中的粘贴代码处理
3. 明确提醒用户：本次迁移以当前粘贴版本为准

## 怎么判断代码范围

无论是哪种输入，都先提炼：

- 真正需要迁移的代码范围
- 这是完整脚本、局部片段，还是单个 kernel
- 是否已经包含调用入口、测试代码、shape/dtype 信息

如果用户只说“迁移这一段”，优先按下面顺序判断：

1. 用户当前高亮或贴出的代码
2. 同一函数 / kernel / wrapper 的完整定义
3. 为了让代码可运行所必需的最小上下文

不要默认把整文件都重写。

## 缺少上下文时的补齐顺序

如果输入信息不完整，按这个顺序处理：

1. 先从现有代码里推断
2. 再用最小合理假设补齐验证脚本
3. 最后才向用户追问必须的信息

优先补齐这些内容：

- 输入张量 shape
- dtype
- device 假设
- 是否存在 wrapper / 调用入口
- 是否要按原脚本风格组织成单文件

## 极小片段的默认策略

如果用户既没有给文件路径，也没有给完整上下文，只贴了极小片段：

- 仍然先迁移片段本身
- 再基于合理假设给出最小验证脚本
- 在结果里明确哪些 shape / dtype / 调用方式是假设值

不要因为输入不完整就停止在泛泛建议。
